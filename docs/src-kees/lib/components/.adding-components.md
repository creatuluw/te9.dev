# Adding New Components - Guide & Checklist

This guide outlines the process for adding new components to the component library, following established patterns and best practices.

## Overview

When adding a new component, you need to:
1. Create the component file (`.svelte`)
2. Create documentation file (`.md`)
3. Export from `index.ts`
4. Add to showcase page (`/cmp`)
5. Follow styling and naming conventions

---

## File Structure

### Component Files Location
All components should be created in: `src/lib/components/`

### Required Files
- `ComponentName.svelte` - The component implementation
- `ComponentName.md` - Documentation (props, examples, usage)

### Optional Files
- `ComponentName.types.ts` - TypeScript types/interfaces (if complex types need to be exported)

---

## Component Creation Checklist

### 1. Create Component File (`ComponentName.svelte`)

- [ ] **File naming**: Use PascalCase (e.g., `ButtonGroup.svelte`, `SearchInput.svelte`)
- [ ] **Script setup**: Use Svelte 5 runes (`$props()`, `$state()`, `$derived()`, etc.)
- [ ] **Props interface**: Define TypeScript interface with all props
- [ ] **Default values**: Set sensible defaults for optional props
- [ ] **Slot support**: Use `<slot />` for content if appropriate
- [ ] **Type exports**: Export any types/interfaces that should be used externally

#### Props Pattern
```svelte
<script lang="ts">
	interface Props {
		variant?: 'default' | 'secondary';
		size?: 'default' | 'sm' | 'lg';
		disabled?: boolean;
		class?: string;
		onclick?: (event: MouseEvent) => void;
		// ... other props
	}

	let {
		variant = 'default',
		size = 'default',
		disabled = false,
		class: className = '',
		onclick,
		...restProps
	}: Props = $props();
</script>
```

#### Styling Guidelines
- [ ] **Design system**: Use zinc color palette (`zinc-900`, `zinc-600`, `zinc-200`, etc.)
- [ ] **Tailwind classes**: Use Tailwind utility classes
- [ ] **Avoid `@apply`**: Prefer inline classes or regular CSS (some Tailwind utilities don't work in `@apply`)
- [ ] **Fonts**: Use `font-aspekta` for headings/buttons, `font-inter` for body
- [ ] **Spacing**: Use consistent spacing (Tailwind spacing scale)
- [ ] **Borders**: Use `rounded-lg` for cards, `rounded-sm` for inputs
- [ ] **Shadows**: Use `shadow-xs` for subtle elevation, `shadow-sm` for hover states
- [ ] **Responsive**: Consider mobile-first responsive design

#### Common Patterns
- Use `class:className={condition}` for conditional classes
- Use `{...restProps}` to pass through HTML attributes
- Use `class={className}` for additional custom classes
- For global styles (when needed), use `:global()` selector

### 2. Create Documentation File (`ComponentName.md`)

- [ ] **File naming**: Match component name with `.md` extension
- [ ] **Structure**: Include required sections (see template below)
- [ ] **Props table**: Document all props with types and defaults
- [ ] **Examples**: Include code examples for common use cases
- [ ] **Variants**: Document all variants if applicable
- [ ] **Sizes**: Document all sizes if applicable
- [ ] **Accessibility**: Note accessibility features if applicable

#### Documentation Template
```markdown
# ComponentName Component

Brief description of what the component does and when to use it.

## Props

| Prop | Type | Default | Description |
|------|------|---------|-------------|
| `propName` | `type` | `default` | Description |

## Variants

### Default
Description of default variant:
```svelte
<ComponentName>Content</ComponentName>
```

## Examples

### Basic Usage
```svelte
<ComponentName onclick={() => console.log('clicked')}>
  Content
</ComponentName>
```

### With Custom Classes
```svelte
<ComponentName class="my-custom-class">
  Custom Styled
</ComponentName>
```
```

### 3. Export from `index.ts`

- [ ] **Add export**: Add component to exports in `src/lib/components/index.ts`
- [ ] **Named export**: Use `export { default as ComponentName } from './ComponentName.svelte';`
- [ ] **Type exports**: If component exports types, add them to type exports section

#### Export Pattern
```typescript
// In index.ts
export { default as ComponentName } from './ComponentName.svelte';

// If types need to be exported:
export type { ComponentProps, ComponentType } from './ComponentName.svelte';
```

### 4. Add to Showcase Page (`src/routes/cmp/+page.svelte`)

- [ ] **Import component**: Add to imports at top of file
- [ ] **Add to TOC**: Add entry to `tocItems` array
- [ ] **Create section**: Add showcase section with examples
- [ ] **Multiple examples**: Show different variants, sizes, states
- [ ] **Interactive examples**: Include interactive examples if applicable

#### Showcase Section Template
```svelte
<!-- ComponentName Component -->
<section id="component-name" class="space-y-6">
	<div class="space-y-2">
		<h2 class="text-2xl font-semibold text-zinc-900 font-aspekta">ComponentName</h2>
		<p class="text-zinc-600">Brief description of the component.</p>
	</div>
	<div class="border border-zinc-200 rounded-lg p-6 bg-white space-y-6">
		<div class="space-y-3">
			<h3 class="text-sm font-medium text-zinc-700">Basic Usage</h3>
			<ComponentName>Example</ComponentName>
		</div>
		<!-- More examples... -->
	</div>
</section>
```

#### TOC Entry
```javascript
const tocItems = [
	// ... existing items
	{ id: 'component-name', label: 'ComponentName' },
	// ... rest of items
];
```

### 5. Testing & Validation

- [ ] **Linting**: Run linter and fix any errors (`read_lints` tool)
- [ ] **Visual testing**: Check component in browser at `/cmp`
- [ ] **Props validation**: Test all props work as expected
- [ ] **Responsive**: Test on different screen sizes
- [ ] **Accessibility**: Verify keyboard navigation and screen reader support
- [ ] **Edge cases**: Test with empty content, long text, etc.

## Design System Conventions

### Colors
- **Primary text**: `text-zinc-900`
- **Secondary text**: `text-zinc-600`
- **Muted text**: `text-zinc-500`
- **Borders**: `border-zinc-200` or `border-zinc-300`
- **Backgrounds**: `bg-white`, `bg-zinc-50`, `bg-zinc-100`
- **Semantic colors**: Use for alerts/status (blue, green, yellow, red variants)

### Typography
- **Headings**: `font-aspekta` (Aspekta font family)
- **Body**: `font-inter` (Inter font family)
- **Code**: `font-pt-mono` (PT Mono font family)
- **Tracking**: `tracking-tight` for headings

### Spacing
- Use Tailwind spacing scale consistently
- Padding in cards: `p-4`, `p-6`, `p-8`
- Gap between elements: `gap-3`, `gap-4`, `gap-6`

### Borders & Radius
- **Cards**: `rounded-lg` with `border border-zinc-200`
- **Inputs**: `rounded-sm` or `rounded-md`
- **Buttons**: Inherits from `.btn` class (uses `rounded-md`)

### Shadows
- **Subtle**: `shadow-xs`
- **Hover elevation**: `shadow-sm`
- **Overlays**: `shadow-lg` or `shadow-xl`

---

## Common Pitfalls to Avoid

1. **Don't use `@apply` with pseudo-class variants** (e.g., `first:`, `last:`)
   - Use regular CSS properties instead
   - Or use `:global()` selectors with standard CSS

2. **Don't forget to export types** if they're used externally

3. **Don't skip documentation** - it's essential for component usage

4. **Don't use inline styles** - use Tailwind classes or `<style>` blocks

5. **Don't forget the `button-group-item` pattern** if component needs special styling when used in groups

6. **Don't mix design systems** - stick to zinc palette and established patterns

## Quick Reference

### Component Location
```
src/lib/components/
├── ComponentName.svelte    ← Component implementation
├── ComponentName.md         ← Documentation
└── index.ts                 ← Update exports here
```

### Showcase Location
```
src/routes/cmp/+page.svelte  ← Add showcase examples here
```

### Import Pattern
```svelte
<script>
  import { ComponentName } from '$lib/components';
  // or
  import ComponentName from '$lib/components/ComponentName.svelte';
</script>
```

## Additional Notes

- **TypeScript**: All components should be fully typed
- **Accessibility**: Consider ARIA attributes where appropriate
- **Performance**: Use `$derived()` for computed values, avoid unnecessary reactivity
- **Consistency**: Follow patterns from existing components
- **Documentation**: Keep docs up-to-date with component changes

---

## Final Checklist

Before considering a component "done", verify:

- [ ] Component file created with proper naming
- [ ] Documentation file created with examples
- [ ] Exported from `index.ts`
- [ ] Added to showcase page at `/cmp` including the TOC entry
- [ ] All props work as expected
- [ ] Styling matches design system
- [ ] No linting errors
- [ ] Tested in browser
- [ ] Responsive design verified
- [ ] Accessibility considered
